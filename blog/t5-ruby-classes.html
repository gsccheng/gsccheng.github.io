Create Your Own Island Today! It's easy with a Ruby Class!

There are many reasons why one wants their own island. A few of the popular ones are:

  1. Get away from zombies
  2. Get away from zombies
  3. and Get away from zombies

So what do you do if for all these great reasons you realize you don't have enough money to get your own island? Well you create one with Ruby using classes of course. It's easy and basically free.

Classes provide a blueprint for you to create objects--in this case an island. What that means is that we can create an island object called zombieless_island class called Island, and if we can create one by calling

zombieless_island = Island.new(2)

"This is confusing, what is the 2 there for?!" you may wonder. Here we are passing an argument to initialize the class. Having such an argument is not necessary and depends on how the class is set up. So, let's dive into how classes are set up!

I think the first thing you should understand are the declaration of variables; classes can use up to 4 different types and are described below. I will describe methods afterwards so don't worry if you run across that words and don't understand the application.



Variable type How it's denoted When do we use it? Example of usage
Instance Variable "@" in front of the name of letters, numbers, or underscore ( _ ) characters When you want a variable accessable for your instance of the object.
When you want to use a variable in multiple methods.		 Diameter of island
Number of defensive cannons installed on island Number of zombies on island
Local Variable Must use letters, numbers, or underscore ( _ ) character. By convention your local variable name should not start with a capital letter. Use this for variables that you would only call within the same method that is it defined in. Hence, this variable is described as local. Say you want to create a method that searches for and prints a report of where any zombies may be on the island. It may use a loop or enumerable to iterate (repeatedly search) through each section of the island. We may then want a way to keep track of how many zombies were found in the current section being checked, then print the result. Once we go through all the sections and the variable is assigned to the number of zombies in the last section, we won't use that information any more for other methods. Hence, there is no point getting it mixed up with other important variables that needs to be accessed outside of this method.
Class Variable "@@" in front of the name of letters, numbers, or underscore characters. When you want to set a variable to be used for all instances of the class When you want to set a maximum height for all islands
Constants Must start with a capital letters. The rest should be letters, numbers, and underscores. By convention your constant name should be in all caps Define constants in a class for use within that class. Although you can change them, the convention is to keep them the same. A variable that sets water as what surrounds the island
Global Variable "$" in front of the name of letters, numbers, and underscore When you want to set a variable to be used in all instances. This is used with caution since it may be difficult to debug where a global variable setting may be coming from. However, if you change a global variable, it gets modified only in one of the server instances for web apps with rails. Use a constant instead if you want it used in all server instances Setting pi to 3.1415

With all these great variables they don't mean anything if we can't do anything with them. That is where methods come in!

Use variables within the definition of the methods. You can also set getter and setter methods (i.e. attr_reader, attr_writer, attr_accessor) at the beginning of the class definition outside of the method definitions. These methods automatically convert instance variables to methods so that they can easily be accessed by operations outside of the object instance
